# Plan: Loading Gate + Campaign Hero Image as Asset

## Context

The audio gate overlay (GameContent.js:2085-2149) currently gates only AudioContext activation â€” a browser gesture requirement. It does zero asset preloading. On slow connections, users click "Enter" and then stare at an empty game while maps, images, and audio download in the background. The download bar in the top nav shows progress, but the user is already "in" with nothing to see or hear.

Additionally, the campaign hero image is downloaded twice: once on the dashboard (raw CSS `backgroundImage`) and again in the game gate (fresh API call â†’ `new Image()`). Neither uses AssetDownloadManager, so the blob cache can't help.

**Goal**: Transform the gate into a genuine loading screen that preloads all active game assets, and migrate `campaign.hero_image` to a proper ImageAsset FK for cross-route caching and user-uploaded hero images.

## Layer 3: Frontend Game Gate â€” Genuine Loading Screen

### Key Insight: Downloads Already Happen Behind the Gate

`MapDisplay` (line 1979) and `ImageDisplay` (line 2024) are rendered behind the z-[102] gate overlay â€” they're NOT gated on `isAudioUnlocked`. So map/image downloads already flow through `useAssetDownload` â†’ `AssetDownloadManager`. Similarly, `syncAudioState` fires on WebSocket `initial_state` and downloads audio blobs via `assetManager.download()`.

**The gate just doesn't wait for them to finish.** The fix is gating the CTA on download completion, not duplicating download logic.

### 3.1 Hero Image via AssetDownloadManager
**File**: `rollplay/app/game/GameContent.js` (lines 570-598)

Currently: fetches campaign metadata â†’ `new Image()` preload â†’ sets `heroImageReady`.

Change the campaign metadata effect to:
- Store `hero_image_asset` info from the response (asset_id, s3_url, file_size)
- Remove the `new Image()` preload logic and `heroImageReady` state

Add `useAssetDownload` for the hero image:
```javascript
const { blobUrl: heroBlobUrl, ready: heroReady } = useAssetDownload(
  campaignMeta?.heroImageAsset?.s3_url,
  campaignMeta?.heroImageAsset?.file_size,
  campaignMeta?.heroImageAsset?.asset_id
);
```
- If the user came from dashboard, this returns instantly from cache (same `asset_id`)
- If direct navigation to /game, it downloads and tracks via the progress system
- For preset `hero_image` (no asset): fall back to direct URL (no caching needed)

### 3.2 Gate Readiness Logic
**New file**: `rollplay/app/game/hooks/useGatePreload.js`

A hook that aggregates readiness signals â€” it doesn't trigger downloads (they're already happening), it just answers "is everything loaded?"

```javascript
export function useGatePreload({ campaignMeta, initialDataLoaded, wsInitialStateReceived, isAudioUnlocked }) {
  const progress = useAssetProgress();

  // All data sources have arrived and had a chance to queue their downloads
  const dataSourcesReady = !!campaignMeta && initialDataLoaded && wsInitialStateReceived;

  // All queued downloads are complete (or lingering)
  const downloadsComplete = dataSourcesReady && !progress.loading;

  return { ready: downloadsComplete, ...progress };
}
```

**Why `dataSourcesReady` matters**: Without it, there's a window between gate render and REST/WS responses where `useAssetProgress()` returns `loading: false` (nothing queued yet), which would falsely show "ready".

Required new state flags in GameContent.js:
- `initialDataLoaded` â€” set `true` at the end of `onLoad()` (after activeMap and activeImage are set)
- `wsInitialStateReceived` â€” set `true` when WebSocket `initial_state` handler fires (in `webSocketEvent.js`)

### 3.3 Gate UI Transformation
**File**: `rollplay/app/game/GameContent.js` (lines 2085-2149)

Replace the current gate with a two-phase UI:

**Phase: Loading** (`!ready`)
- Campaign title + description (available early from campaignMeta)
- Hero image background (from `heroBlobUrl` or preset URL, may fade in)
- **Progress bar** from `useGatePreload` â€” reuse the same visual style as the nav download bar:
  - `{completedCount}/{totalCount} assets`
  - Byte-level progress bar: `width: (loadedBytes / totalBytes) * 100%`
- "Click to enter" text hidden or dimmed/disabled
- Connected players list (keep existing)

**Phase: Ready** (`ready`)
- Same layout
- Progress bar completes and fades
- "Click to enter" becomes visible/active with a subtle entrance animation (pulse or glow)
- Click triggers existing `handleEnterSession` (audio unlock + auto-seat)

The outer gate condition remains `{!isAudioUnlocked && (...)}`. The click handler remains `handleEnterSession`. The only change is that the clickable area / CTA is gated on `ready`.

### 3.4 Overlay GIF Preloading

Overlay GIFs (`/cine/overlay/film-grain.gif`, etc.) are local files, not S3 assets. `ImageDisplay` already preloads them via `Promise.all(urls.map(url => new Image()))` (lines 89-119). Since `ImageDisplay` is mounted behind the gate, this preloading already happens.

If `ImageDisplay` is NOT mounted (e.g., `activeDisplay === 'map'`), overlays aren't relevant to the current view and don't need preloading.

No changes needed here.

## Implementation Order

### Phase 1: Backend (backward-compatible, deploy first)
1. Model + relationship (must exist before autogenerate can detect the change)
2. Alembic autogenerate migration â€” `docker exec api-site-dev alembic revision --autogenerate -m "add hero_image_asset_id to campaigns"`
3. Aggregate + create/update methods
4. Schemas (request + response)
5. Commands â€” pass through new field
6. Repository â€” read/write new field
7. Endpoints â€” resolve FK to presigned URL in response builders

**Result**: API returns new `hero_image_asset` field alongside existing `hero_image`. Frontend ignores unknown fields â€” fully backward compatible.

### Phase 2: Dashboard hero image (depends on Phase 1)
1. `useHeroImage` hook â€” branching logic for asset vs preset
2. Campaign cards â€” swap raw CSS for hook-backed `backgroundImage`
3. Hero image picker â€” presets + library sections
4. Mutation updates â€” send `hero_image_asset_id`

**Result**: Users can pick library images as hero images. Dashboard caches S3 hero images in AssetDownloadManager.

### Phase 3: Game loading gate (depends on Phase 1, benefits from Phase 2 cache)
1. Campaign metadata effect â€” store `hero_image_asset` info
2. Hero image via `useAssetDownload` â€” cache hit from dashboard
3. `initialDataLoaded` + `wsInitialStateReceived` flags
4. `useGatePreload` hook
5. Gate UI â€” progress bar + gated CTA

**Result**: Gate shows real loading progress. CTA appears only when all assets are ready. Hero image is instant if user came from dashboard.
